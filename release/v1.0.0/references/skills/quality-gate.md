# Quality Gate — 质量门控

## Overview

报告可以写得有深有浅，但不能缺少关键检查项。质量门控是输出前的最后一道防线——检查报告是否满足基本质量要求。

门控逻辑：**缺一项就退回修改，不通过不输出。**

## When to Use

- 每次输出报告（任何轮次、任何版本）之前

## The Process

### Step 0: 加载对应 Skill 的 Verification 清单

本 gate 是顶层检查。每个具体 skill 在 Process 末尾有自己的 Verification 清单（如 `deep-tech-dd-analysis` 有 10 项验收条件）。执行本 gate 前，先确认使用了每个 skill 的 Verification。

---

### 质量检查清单（10 项）

**Checklist:**
- [ ] 1. **Claim验证完整性** — 每个核心技术 claim 是否有独立的物理/工程定量验证（公式+数值计算），而非仅定性评价？
- [ ] 2. **工程迁移鸿沟** — 是否明确分析了从原理验证到工程产品的具体鸿沟和未验证环节？
- [ ] 3. **逐人团队评估** — 是否对每个核心成员做了能力匹配度评判（相关/部分相关/不相关），而非笼统说"团队不错"或"团队不行"？
- [ ] 4. **独立竞品发现** — 是否主动调查并指出了 BP 遗漏的竞品或技术替代方案？
- [ ] 5. **定量评分** — 评分是否带小数位（如 3.5/5），体现判断精度而非四舍五入？
- [ ] 6. **优先级区分** — 待验证问题是否按 P0/P1/P2 分了优先级？
- [ ] 7. **供应链具体到供应商级别** — 供应链分析是否有具体供应商名称和产能判断，而非笼统说"有风险"？
- [ ] 8. **评分与结论自洽** — 评分是否与正文中具体分析的 flag 数量和严重程度匹配？
- [ ] 9. **概述页** — 报告正文第一页是否是核心结论的概述页（Claim 判断表 + 概要评级）？
- [ ] 10. **非 BP 复述** — 报告是否独立分析了技术逻辑/行业逻辑/市场逻辑，而非只是转述 BP 的观点？

---

### 核查方法

逐项检查。任一未通过项必须在报告中补充对应内容后再输出。如果某个维度因为项目情况天然不适用（如 BP 中没有供应链信息因此无法做供应商级别分析），需在报告中明确标注"该维度因缺少信息未评估"并记录到信息缺口清单，不能跳过不做说明。

## Red Flags

1. 报告看起来"太完美"——大概率是BP的复述
2. 所有评价都是正面/中立，没有负面发现
3. 评分全是整数（4/5、3/5 而非 3.5/5、2.5/5）——判断置信度没有体现
4. 报告中大量"团队表示…""BP中宣称…"的句式，没有独立的"经调查发现…"段落

## Verification

- [ ] 所有 10 项检查清单逐项自检通过
- [ ] 不可用且注释了"因缺少信息未评估"的项不超过 2 项（超过 2 项说明信息缺口太大）
- [ ] 自检结果已记录在报告中（可在附录中简注）
