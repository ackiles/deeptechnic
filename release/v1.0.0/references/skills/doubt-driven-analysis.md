# Doubt-Driven Analysis — 质疑驱动分析

## Overview

一个自信的答案不等于正确的答案。长时间在同一上下文中工作，假设会悄悄变成"事实"。质疑驱动分析是把一个 fresh-context 的审查者——偏向于**证伪**而非证实——引入每个非平凡的技术 claim 决定之前。

这不是法务合规审查，也不是技术评估报告。这是：**在每个 claim 被接受之前，找一个人（或一个模型）从零开始专门找茬。**

## When to Use

- **第一轮初评** — 对 BP 中最核心的 3-5 个技术 claim 执行质疑循环
- **团队回复后** — 团队给出了新的数据或解释，对这些新信息做质疑
- **任何感觉"这听起来太完美了"时** — 直觉是红旗信号

**NOT for：** 机械操作、明确引用标准/论文的陈述、用户明确要求跳过验证的场景

## The Process

### Step 1: CLAIM — 摘出要审的表述

用两到三行精确写出待审核心 claim：

```
CLAIM: "团队声称15um像元可以实现与30um像元同等的NETD 20mK"
WHY THIS MATTERS: 像元面积缩小到1/4，理论上NETD会劣化4倍
```

如果写不出这么紧凑的 claim，说明这是个"感觉"而不是"决定"——先澄清再质疑。

### Step 2: EXTRACT — 最小可审单元

给审查者需要的不是整份 BP，而是**工件 + 合约**（artifact + contract）：
- 技术参数：相关的物理公式 + BP 宣称数值 — 不是全文
- 创新点：方案描述 + 对比基准 — 不带团队的解释
- 断言：声明 + 支撑它的证据 — 与 Step 1 的判断分开放

**剥离推理过程。** 如果你把结论也给了审查者，你得到的是对你结论的确认。

### Step 3: DOUBT — 发起对抗式审查

审查者的 prompt **必须是对抗性的**。提问方式决定了答案。

```
对抗式审查。找出这个工件有什么问题。
假设作者过度自信。寻找以下类型的问题：
- 未声明的假设
- 未处理的边缘情况
- 隐藏的耦合或共享状态
- 合约可能被违反的方式
- 现有约定被破坏的可能性
- 意外输入下的失效模式

不要验证。不要总结。找出问题，或者明确声明经彻底检查后未发现问题。

工件（ARTIFACT）：<粘贴相关技术描述>
合约（CONTRACT）：<粘贴验收条件>
```

**只传 ARTIFACT + CONTRACT。不要传 CLAIM。** 给审查者你的结论会让人倾向于同意。

### Step 4: RECONCILE — 对账

将审查者的发现分类：
- **真实缺陷** — 确实存在的问题，需要修改 claim 或标记为风险
- **误报** — 审查者理解有误的，忽略
- **边界案例** — 现有方案中未明确，需团队回答的

### Step 5: STOP — 满足停止条件

停止条件（满足任一即可）：
1. 连续 3 个 claim 审查后没有新的真实缺陷发现
2. 所有发现的缺陷都被归类和处理
3. 用户明确说"够了"

## Verification

- [ ] 至少对 3 个核心 claim 执行了完整的 5 步质疑循环
- [ ] 每个 CLAIM 写成了两行以内的精确表述
- [ ] EXTRACT 时剥离了推理过程（没传结论）
- [ ] DOUBT 使用了对抗式 prompt
- [ ] RECONCILE 将发现分为了真实缺陷/误报/边界案例三类
- [ ] 发现的问题已标记到评估报告对应章节
