# Iteration Merge — 同轮次内迭代信息融合

## Overview

用户阅读报告初版 → 提几个问题 → 我回答 → 用户说"那按我们讨论的修改一下报告"。这个过程是**信息丢失的高发环节**——我回答中暴露的详细分析逻辑如果没有被吸收进更新报告，用户会看到"前后不一致"或"重复质问"的效果。

本 skill 确保问答过程产生的新信息不会丢失。

## When to Use

- 用户在同一轮次中要求更新/修改报告时

## The Process

### Step 1: 回溯问题-回答链条

回顾本轮对话中用户的每个问题，以及我的每个回答。识别每个回答中新增的关键信息——这些信息需要和报告原有内容做对比，分成 4 类：

| 类型 | 特征 | 处理方法 |
|------|------|----------|
| **修正型** | 新信息显示原结论错了 | 直接修改原章节 |
| **补充型** | 新信息在原结论之外增加了新维度 | 追加分析段落 |
| **强化型** | 新信息为原结论提供了更强证据 | 在原论证中加入新推理 |
| **额外判断** | 问答中产生了超出原报告范围的判断 | 新增独立分析段落 |

### Step 2: 融合新旧信息

将分类后的新信息写入报告对应章节：
- 修正型 → 标注"此结论已更新（基于XX讨论）"
- 补充型 → 在对应章节末尾追加，标注"经讨论补充XX"
- 强化型 → 保持原结论，加入新的支撑证据
- 额外判断 → 新增为独立分析段落（标注来源）

### Step 3: 编写修改逻辑说明

在报告末尾附修改逻辑说明表：

```markdown
### 本轮修改逻辑说明

报告版本：v1 → v2（2026-06-15 更新）

| 修改位置 | 修改类型 | 修改内容 | 修改原因 | 新信息源 |
|----------|----------|----------|----------|----------|
| 第三章.2节 | 修正 | NETD判断从🟡→❌ | 用户指出BP数据与行业基准不一致 | 用户追问+web_search核实 |
| 第五章.1节 | 补充 | 新增对ZZ竞品分析 | 用户提出BP遗漏的关键竞品 | technology-news-search检索 |
| 附录问题清单 | 追加 | 新增2个P0级问题 | 讨论暴露了新的风险点 | (用户直接提出) |

修改统计：修正 X 处 / 补充 Y 处 / 新增 Z 处 / 删除 0 处
```

### 例外规则

- 如果用户明确要求"只改某一部分，其他不动"，则在修改逻辑说明中标注"局部修改，未涉及部分与原版一致"
- 如果新信息推翻了整个报告核心判断，先向用户确认后再全面修改
