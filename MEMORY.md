## 核心技术方法论

### 初评"两个问题"约束
第一轮只聚焦两个问题：
1. 技术是否**真正先进且可行**？
2. 团队是否**有能力做出来**？
- 市场/供应链/竞品分析后移到后续轮次
- 违反此约束会导致质量维度打分偏低

### 独立工程分析
- 每一版报告必须从物理公式推算开始，不能转录BP宣称
- 识别BP声称中的物理矛盾（如"15μm像元 20mK NETD"需解释守恒机制）
- 逐项标注不确定性和置信度
- P0/P1/P2优先级分组输出

### 团队逐人审计
- 姓名、学术背景、公开渠道可验证性、与项目能力的匹配度
- 核心产出声明必须有DOI/论文编号/预印本可交叉验证
- "submit to" ≠ "published" ≠ "accepted"
- 结构性缺口分析（团队缺什么方向的工程能力）

### BP反合理化
- 识别BP话术：稻草人攻击、时间偏移比较、选择性遗漏竞品
- 标准化反驳话术，优先级P0/P1/P2
- 已沉淀15条BP claim-rebuttal规则在 `references/data/bp_claims.json`

## 系统规则记忆

### 工作流
- **4轮尽调：** Round 1信息收集→Round 2团队反馈→Round 3专家访谈→Round 4综合结论
- **5阶段会话：** Phase 0项目启动→Phase 1初评→Phase 2反馈分析→Phase 3访谈→Phase 4综合
- **10项质量门控：** 每个报告必须通过 `check_quality_gate.py` 全部检查

### 工具调用
| 工具 | 用途 | 降级方案 |
|------|------|---------|
| sn-deep-research | 核心研究编排器 | - |
| scholarclaw | 学术论文搜索 | web_fetch直搜arXiv/PubMed |
| technology-news-search | 产业新闻 | 垂直领域可能无效 |
| web_fetch/browser | 特定URL内容 | - |

### 关键约束
- **项目白纸原则：** 每个项目独立对话，不跨对话引用上下文
- **分域隔离：** 系统优化对话不讨论具体项目
- **认知刷新：** 团队反馈不是"批改作业"，是刷新整个认知模型
- **信源缺口提醒：** 发现关键信息缺失时简短提示，不强行阻断

## 已知经验教训

1. **第一版报告不要转述BP** — 用户反馈"是BP综述而非技术分析"，重写为独立工程分析
2. **scholarclaw可能不可用** — HTTP 308/404 时有发生，备选方案为 web_fetch
3. **technology-news-search对超垂直医疗主题无效** — RSS源不覆盖MedTech Dive/MassDevice
4. **中文引号问题** — python-docx 比 docx-js 更稳定，避免 Unicode 引号导致的 JS 语法错误
5. **web_fetch 被墙** — Wikipedia/PubMed/ScienceDirect 可能被reCAPTCHA或JS验证拦截
6. **教授任CEO有条件接受：** 有创业经历和产业化思维的教授不是问题
