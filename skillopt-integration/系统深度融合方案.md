### 3.1 工作流程规则 → agent-skills 格式转化

将 [02-workflow-rules.md](file:///Users/ackiles/Documents/AI-Space/专家/02-workflow-rules.md) 拆分为独立 skill 文件，每个 skill 遵循统一格式：

```markdown

aliases: [核心claim验证, 技术分析]

# 核心技术分析

## Overview

对项目BP中的核心技术 claim 逐一做独立验证，验证需要有物理公式
和数值计算支撑，而非定性描述。

## When to Use

- 第一轮初评：收到BP后，对核心技术部分做系统验证
- 第二轮复核：团队回复后，对修正后的技术指标做二次验证
- 用户明确要求"深入分析XX技术点时"

## Process（步骤 Checklist）

### Step 1: CLAIM 提取
- [ ] 阅读 BP 全部技术章节，提取 4-6 个最核心的技术 claim
- [ ] 每个 claim 用一句话写清楚（格式：「宣称：XXX」）
- [ ] 标注每个 claim 的来源章节和页码

### Step 2: 物理定量验证
- [ ] 对每个 claim，找到对应的物理/工程公式
- [ ] 代入实际数值做反向计算
- [ ] 判断公式中各参数之间是否存在矛盾约束

### Step 3: 工程迁移鸿沟分析
- [ ] 原理验证的边界条件（实验室/理想环境 → 实际现场）
- [ ] 列出每个工程环节的已知瓶颈
- [ ] 团队在该环节的经验状态

### Step 4: 判断输出
- [ ] 每个 claim 给出 ✅/🟡/❌ 判断
- [ ] 汇总未被充分验证的假设清单

## Rationalizations（常见借口与反驳）

| BP常见话术 | 标准反驳 |
|-----------|---------|
| "我们的指标达到了国际领先水平" | "请提供第三方测试报告和量化对比数据" |
| "这个参数还在优化中" | "没有初步测试数据，无法评估基础可行性" |
| "我们的技术路径是独创的" | "请提供专利检索结果和竞品对标分析" |
| "我们的算法比所有竞品都好" | "请给出具体竞品型号、参数、测试条件" |
| "我们的技术可以广泛应用于XX领域" | "请给出至少一个该领域的客户验证或 POC 结果" |

## Red Flags

- BP 中所有指标都是整数（无小数范围值 → 无实测数据）
- 只描述功能/效果，不给技术指标
- 声称多项极限指标同时成立（物理上不太可能）

## Verification

- ☐ 至少 3 个 claim 完成物理定量验证
- ☐ 每个 claim 有对应的公式或工程约束分析
- ☐ 列出全部待验证假设（含优先级）
- ☐ 是否发现 BP 中隐藏的物理矛盾
```

## 四、三者的角色分工与依赖关系

| 层面 | 组件 | 核心职责 | 依赖 |
|------|------|----------|------|
| **生产 (Production)** | Deeptechnic | 执行尽调、输出报告、收集反馈 | 依赖 agent-skills 格式的方法论 |
| **方法论 (Methodology)** | agent-skills | 定义 skill 标准格式、版本管理、生命周期 | 独立的，但输出被 Deeptechnic 使用 |
| **优化 (Optimization)** | SkillOpt | 从反馈数据中自动学习、优化 skill 文档 | 依赖 agent-skills 格式（数据格式标准）和 Deeptechnic 的反馈数据 |

**依赖关系图：**

```
Deeptechnic ──使用──→ agent-skills 格式的技能文档
     │                        │
     │ (产出评估数据)          │ (提供优化目标)
     ▼                        ▼
  SkillOpt ──优化──→ agent-skills 文档 → Deeptechnic 使用新版
```

## 六、关键设计决策

| 决策 | 选项 A | 选项 B | 选择 |
|------|--------|--------|------|
| skill 文档位置 | 放在当前仓库 | 放在独立 repo | **当前仓库**（方便开发和测试，后期成熟可拆） |
| 版本管理 | Git + 手动 diff | SkillOpt best_skill.md 自动版本 | **两者都要** — Git 管语义版本，SkillOpt 管参数版本 |
| 质量门控 | 统一在一个 quality-gate.md | 每个 skill 各自门控 | **统一 gate + 各自 verification** |
| Persona 是否是 skill | 是（独立的 skill 文件） | 否（单独的 agents/ 文件） | **单独的 agents/ 文件**（agent-skills 的做法） |
| 首次优化目标 | 优化所有 skill 同时 | 只优化 1-2 个 skill | **先优化 deep-tech-dd-analysis + anti-rationalization** |

## 七、一次完整迭代的检查清单

当完成一轮"评估项目 → 用户反馈 → 自评记录 → 触发训练"后：

```markdown
## 第X轮反馈-训练-更新循环

### 输入
- [ ] 完成的项目评估（至少1-3个）
- [ ] 对应每个项目的用户反馈记录
- [ ] 自评10维评分表

### 训练
- [ ] 数据已追加到 skillopt-integration/data/deep_tech_dd/
- [ ] 数据量 ≥ 20 条 → 运行 SkillOpt 重训练
- [ ] 数据量 < 20 条 → 标记为"+N条，继续积累"

### 审查
- [ ] 查看 best_skill.md vs 当前技能文档的 diff
- [ ] 确认优化方向与用户反馈一致
- [ ] 解决了历史报告中的哪些问题

### 合并
- [ ] 有意义的编辑 → 手动合入 skills/ 对应文件
- [ ] 更新 02-workflow-rules.md 的版本号
- [ ] 本次循环记录追加到 .learnings/LEARNINGS.md
```
